## 经验教训

1. **10个工具里程碑**：这是项目的第10个工具，意味着工具集已经形成规模效应。从单个工具到10个工具，交叉链接管理的工作量从每次更新1个文件增加到每次更新10个文件。建议考虑通过模板化或预处理方式来简化未来的维护。

2. **不同工具结构的兼容性**：项目中的工具由多次独立运行构建，产生了3-4种不同的 HTML 结构（直接 nav、pill-group nav、info-section 链接列表、info-list 链接列表等）。这给批量更新带来了很大挑战。后续新工具应统一采用最新的模板格式（three-context: nav + info-list + footer-sep）。

3. **patch replace_all 的风险**：使用 `replace_all=True` 要极其谨慎——它会替换文件中所有匹配的子串，包括页面标题、导航文字、甚至其他不相关部分的意外匹配。这次差点毁掉 README，以后宁愿写完整文件也不要使用 replace_all。

4. **颜色选择器是设计领域高频工具**：竞品分析显示几乎所有主流工具站都有颜色选择器，说明这是稳定的流量入口。功能完整的颜色选择器（可视化取色 + 多格式互转 + 配色方案）可以在竞争中形成差异。

5. **项目规模已达 30+ 文件**：10个工具 + 首页 + README + 9篇 SEO 文章 + 9个运行日志 → 约 30 个文件，约 250KB。需要更谨慎的变更管理。

6. **Python 脚本 vs 手动 patch**：对于跨文件的批量更新（如交叉链接），Python 脚本比逐个 patch 高效得多，但需要处理 HTML 结构的多样性。结合使用：脚本做批量操作，patch 做精确修复。
