## 经验教训

1. **URL 编码解码器是开发者的高频需求**：从搜索和竞品分析看，URL 编码解码工具在开发者工具站中属于高频使用类别。几乎所有主流工具站都有这个工具。

2. **三种操作模式设计合理**：编码、解码、批量处理三种模式覆盖了绝大部分使用场景。分段按钮切换比在一个页面混合展示所有功能更清晰。

3. **灵活选项提升实用性**：编码所有字符、空格用+替代、UTF-8切换这三个选项覆盖了所有常见的 URL 编码变体。不常需要但需要时非常关键。

4. **patch 工具的局限性**：当文件有复杂的不一致缩进时，patch 容易出现格式错乱。需要在 patch 后验证。对于时间戳转换工具发现了一个 bug — footer 原来链接到自身（`../timestamp-converter/`），趁这次修复了。

5. **9个工具的交叉链接复杂度**：每次构建新工具需要逐个修改8个现有工具的3个上下文区域（复杂工具约24次 patch 操作）。如果使用更高级的方法（模板渲染或 include 预处理）会大幅减少工作量，但纯静态 HTML 的方案不需要额外工具链。

6. **项目规模持续增长**：9个工具共涉及 25+ 个文件，约 200KB+ 的 HTML。每次更新需要谨慎管理交叉链接，验证所有工具的链接完整性。
