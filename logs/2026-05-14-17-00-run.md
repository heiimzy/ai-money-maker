## 经验教训

1. **单位换算器已提前构建完成**：当第8次运行时发现单位换算器已经存在于文件系统中（有完整的30KB index.html），说明前一次运行已构建但未提交。需要更注意检查 git status 确认构建状态。

2. **早期工具结构差异**：最早的两个工具（字数统计、UUID生成器）只有 tools-nav 区域有交叉链接，没有 info-section 链接列表和 footer 分隔列表。它们的 footer 是简单的版权声明。这些工具是在建立完整三上下文规范之前构建的，可以接受不需回改。

3. **搜索稳定性问题**：DuckDuckGo 对中国地区的搜索查询经常超时。用 Python DDGS 加 time.sleep(3) 延迟后，第一个查询能返回结果但第二个开始超时。这限制了研究环节的质量。

4. **GitHub推送依赖凭证**：没有可用的 GitHub PAT，无法通过 git push 或 API 推送更新。需要用户提供一个 repo-scoped Classic PAT（ghp_开头）才能部署到 GitHub Pages。

5. **项目持续增长**：8个工具共涉及 20+ 个文件，每次更新需要管理大量交叉链接。现在的 patch 替换策略比全文件编辑高效，但需要确保替换上下文的唯一性。
